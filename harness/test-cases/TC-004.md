## Description
End-to-end test of the Treasure Trove storefront Product Detail Page (PDP) against the mock data layer. Covers the full buy-box flow at `/products/[handle]` — Breadcrumb, ImageGallery (thumbnail strip, main image, zoom/crossfade), VariantSelector (Material / Size / Finish + Quantity), real-time price update on variant change, StockIndicator (In Stock / Only N left / Out of Stock), AddToCartButton (loading, success, disabled states), MaterialStoryLink (rendered only when story data is present), JSON-LD Product schema in `<head>`, the "About This Piece", Specifications and Care sections, the Reviews section, and the RelatedProducts horizontal carousel using ProductCard. Also validates the three required edge-case routes — Product Found (`halden-brass-pendant-lamp`), Product Not Found (`/products/any-nonexistent-handle` → `notFound()`), and Out of Stock (Polished Silver + M variant) — plus code-standard checks (no raw `<img>`, no `"use client"` on the page, no real Medusa traffic, `data-testid` coverage on every interactive element) and mobile-first responsive layout across 360 / 768 / 1440 px viewports. The page must be a Server Component with cart/variant interactivity isolated to `"use client"` children.

## Preconditions
- `apps/storefront` is running (`pnpm dev --filter=storefront` at `http://localhost:3000`)
- Mock product layer is active — `src/lib/mock/product-detail.ts` (`getProductByHandle`) and `src/lib/mock/material-story.ts` (`getMaterialStory`) are the live implementations; no real Medusa SDK or Payload REST calls are required
- Mock dataset contains at least the `halden-brass-pendant-lamp` product with:
  - 5 images, 3 finishes (Antique Brass, Polished Silver, Matte Black), 3 sizes (S / M / L)
  - The `Polished Silver / M` variant pinned to `inventory: 0, available: false` for the Out-of-Stock flow
  - At least one variant with `inventory ≤ 5` to exercise the "Only N left" low-stock state
  - A `materialStorySlug` populated so the Material Story Link renders
  - 6 related products available via `RelatedProducts`
- Zustand cart store at `src/stores/cart.ts` is wired and exposes `addItemLocal` (mock-safe add)
- Browser in a fresh state — no cookies, no `localStorage` keys from prior sessions
- Browser DevTools open with the Network tab visible (so you can confirm no real `localhost:9000` / Payload traffic)
- Chrome DevTools device toolbar available for mobile / tablet / desktop viewports
- Browser does NOT have `prefers-reduced-motion: reduce` set system-wide (Framer Motion transitions are validated explicitly)

## Notes
- The page is currently wired to `getProductByHandle` and `getRelatedProducts` from `@/lib/medusa` and `getMaterialStory` from `@/lib/payload` — these are the active mock implementations per TASK-026. Verify by reading `apps/storefront/src/lib/medusa.ts` and `apps/storefront/src/lib/payload.ts` before running. If those files have been swapped to a real SDK, this TC must be re-scoped or run against a real seeded backend.
- The `Polished Silver / M` variant is pinned to `inventory: 0, available: false` in the mock specifically to exercise the Out of Stock flow — if a future seed mutates this row, re-pin it before running step 20.
- AddToCart uses `addItemLocal` (mock-safe). The real `addItem` (Medusa-backed) remains in the cart store but must not fire during this TC — confirm via the Network tab.
- Finish colour swatches use inline `style={{ backgroundColor }}` — this is the documented exception to the "no inline styles" rule for data-driven dynamic colours and is NOT a failure.
- The Buy Now button (`data-testid="buy-now-button"`) is rendered but its checkout wiring is out of scope for TASK-026 — clicking it during this TC is informational only; treat any wiring change as a separate ticket.
- Wishlist (`WishlistButton`), Reviews submission flow, and the global Navbar / BottomTabBar cart badge are covered by their own TASK-XXX cases — exercise them here only to confirm they integrate (cart count increments, drawer opens) but do not deep-dive their internal states in this TC.
- Real Medusa product / variant / pricing wiring, server-side cart sync, Payload Material Story content delivery, and ISR webhook regeneration are **out of scope** — those are covered by TASK-027 (Backend), TASK-028 (Integration Testing), and TASK-033 (Security).
- If Framer Motion transitions appear janky or trigger layout shift during step 5, step 18, or step 31, file a P1 bug — TASK-026 explicitly calls out smooth crossfade and carousel transitions as acceptance criteria.
- Use the React DevTools "Components" tab to confirm `app/products/[handle]/page.tsx` is rendered as a Server Component; any `"use client"` leak to the page level is an automatic P0 failure.
