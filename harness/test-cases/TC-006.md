## Description
End-to-end test of the Treasure Trove storefront 4-step checkout flow at `/checkout` against the mock layer (`src/lib/checkout.mock.ts`). Covers the full happy path — Step 1 Address (React Hook Form + Zod with country-aware pincode validation), Step 2 Shipping (radio list of three mock shipping options with carrier, estimated window and price), Step 3 Payment (Razorpay UI for INR / Stripe card element for USD/AED, with a `simulate-failure` QA toggle), Step 4 Confirmation (order ID, itemised order summary, delivery address summary, "Continue Shopping" link) — the `CheckoutStepper` active/completed/pending visual states across steps, the sticky `OrderSummary` sidebar with the promo code input + apply, the two required edge cases (Address Validation server error + Payment Failed with retry), the Back navigation between steps without losing state, and code-standard checks (`"use client"` page, no full-page navigations between steps, no raw `<img>`, `data-testid` coverage on every input/button, zero real Medusa traffic). Also validates responsive layout — sticky sidebar at `lg` and above; order summary stacked above the form below `lg`.

## Preconditions
- `apps/storefront` is running (`pnpm dev --filter=storefront` at `http://localhost:3000`)
- Mock checkout layer is active — `src/lib/checkout.mock.ts` is the live implementation; `mockGetShippingOptions`, `mockInitiatePayment` (with optional `simulateFailure`), and `mockPlaceOrder` are unmodified
- Mock cart seed contains at least two line items (`Artisan Sculpted Vase / Ivory / qty 1` and `Royal Velvet Cushion Set / Mauve / qty 2`); subtotal computes to `₹20,174` based on `MOCK_CART_ITEMS`
- Three mock shipping options are available — Standard (free), Express (₹599), White Glove (₹1,499)
- `apps/storefront/src/app/checkout/page.tsx` is a `"use client"` component (per TASK-042 AC); `layout.tsx` for `/checkout` strips the global Navbar / Footer or applies the checkout-specific chrome (verify by reading `app/checkout/layout.tsx`)
- Browser in a fresh state — no cookies, no `localStorage` keys carrying checkout state from prior runs
- Browser DevTools open with the Network tab visible (so you can confirm no real Medusa traffic and no live Razorpay / Stripe SDK calls)
- Chrome DevTools device toolbar available for mobile / tablet / desktop viewports

## Expected Result
The 4-step checkout behaves end-to-end against the mock layer. AddressStep validates all required fields via React Hook Form + Zod (including country-aware pincode rules — 6 digits for India, 5–10 for others) and exposes an inline server-error fallback. ShippingStep loads the three mock options, requires a selection before enabling Continue, and surfaces a load-error fallback if `mockGetShippingOptions` rejects. PaymentStep conditionally renders Razorpay for INR carts and Stripe for USD/AED carts, with a `simulate-failure` QA path that produces a recoverable "Payment declined" error + Retry button. ConfirmationStep renders an order ID matching `TT-2026-XXXXX`, the itemised summary, the delivery address, and a "Continue Shopping" link to `/products`. The CheckoutStepper accurately renders active/completed/pending visual states across all four steps. The sticky OrderSummary lists the mock cart items, applies the chosen shipping cost in real time, and exposes a promo-code field. Back navigation between steps preserves entered state. All step transitions are client-side (no full-page reload); the page is a `"use client"` component. All inputs and buttons expose `data-testid` attributes; every image renders via `next/image`; the Network tab confirms zero real Medusa, Razorpay, or Stripe traffic across the full flow. The layout is responsive — sticky sidebar at `lg+`, stacked-above-form below `lg`.

## Notes
- `PaymentStep.tsx` ships a `simulate-failure` QA checkbox per the TASK-042 review notes — this is intentional QA tooling, NOT a leak. If it is hidden behind a dev-only flag, expose it before running steps 24–27 (or mutate `mockInitiatePayment`'s default `simulateFailure` argument directly).
- The order ID pattern `TT-2026-XXXXX` is hard-coded in `mockPlaceOrder` (`TT-2026-${10000 + random * 90000}`). Any deviation — different prefix, different digit count, missing year — is a P1 regression.
- The `mockPlaceOrder` hard-codes `currency: 'INR'` in its return — for the USD/AED branch test (step 29–30), the OrderSummary on Confirmation may still show INR formatting unless the mock is also patched to forward the cart's currency. Document the observed behaviour in the Actual Result; treat a mismatch as a P2 noted-but-not-blocking finding.
- `OrderSummary.tsx` was added beyond the original TASK-042 spec to separate sticky-sidebar logic from step components — it is intentional, not scope creep, and is covered by this TC.
- The Razorpay and Stripe payment flows must be **UI-only** — no live SDK fetches. If `js.stripe.com/v3` or Razorpay's checkout JS appears in the Network tab, file a P0 bug — this would imply a live SDK initialisation that TASK-042 explicitly disallows.
- React Hook Form + Zod is the mandated pattern for the AddressStep — no raw `useState` for input values. `useState` is acceptable only for UI flags (current step, payment-method selection, error visibility).
- Pincode formats beyond India / "Other" (e.g. Canadian alphanumeric, UK postcodes) are NOT validated by the current schema — out of scope for this TC; flag in QA notes if the country selector exposes such options but the schema does not enforce them.
- Real Medusa cart sync, server-side address / shipping creation, real payment-provider initialisation, and order persistence are **out of scope** — those are covered by TASK-043 (Backend), TASK-044 (Integration), and TASK-049 (Security).
- Use the React DevTools "Components" tab to confirm that `apps/storefront/src/app/checkout/page.tsx` is a client component (it owns the step state). Any leak of `"use client"` to a server-only file, or any attempt to fetch checkout data in a Server Component, is a P0 bug against TASK-042.
