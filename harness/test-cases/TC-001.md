## Description
End-to-end test of the full storefront authentication flow on the Treasure Trove storefront against the mock auth layer. Covers the complete user journey across all four `(auth)` routes — `/login`, `/register`, `/forgot-password`, `/reset-password` — including: branding and responsive layout, Zod validation on every form, successful registration → `/account`, successful login → `/account`, wrong-password inline error (never a toast), 5-attempt account lockout with 30-minute countdown, "Email already registered" inline error with a link to `/login`, the full forgot-password + reset-password flow with all three token branches (valid, invalid, expired), submit-time token-expiry race, and code-standard checks (no raw `<img>`, no `"use client"` on pages).

---

## Preconditions
- `apps/storefront` is running (`pnpm dev --filter=storefront` at `http://localhost:3000`)
- Mock auth layer (`apps/storefront/src/lib/auth/mock.ts`) is the active implementation — no real Medusa session calls are required
- Mock recognises the following test fixtures (verify in `src/lib/auth/mock.ts`):
  - `valid@test.com` / `Pass@1234` → login success
  - `taken@test.com` → register rejects with `EMAIL_EXISTS`
  - 5 consecutive wrong passwords for `valid@test.com` → lockout with 30-min cooldown
  - Tokens: `valid-token`, `expired-token`, `invalid-token` for `/reset-password`
- Browser in a fresh state — no cookies, no `localStorage` keys from prior sessions
- Browser DevTools open with the Network tab visible (so you can confirm no real backend calls fire)
- Mobile + Desktop viewports available (Chrome DevTools device toolbar)

## Expected Result
The complete authentication flow works end-to-end against the mock layer. Registration succeeds for new users and rejects already-registered emails inline (never as a toast) with a "Sign in instead" link. Login redirects valid users to `/account`. Wrong-password attempts show inline errors below the form; the 5th consecutive failure locks the account with a live 30-minute `MM:SS` countdown and a recovery CTA. The forgot-password success screen is identical for known and unknown emails (no enumeration). All three reset-password token branches — missing, invalid, expired — render distinct recoverable screens. A valid token mounts the new-password form after a "Validating link…" indicator; a successful reset routes the user to `/login` with a success indicator. All forms are built with React Hook Form + Zod, inputs expose `data-testid` attributes, every image renders via `next/image`, pages are Server Components by default, and the layout is mobile-first responsive across 360 × 640 and 1440 × 900 viewports.

## Notes
- All forms must be built with React Hook Form + Zod — confirm there is no `useState` bound to input values. `useState` is acceptable only for UI flags (show/hide password, dialog open/close).
- "Wrong password error inline, not toast" is a hard TASK-002 AC. Any step that surfaces a toast for credential errors is an automatic P0 failure.
- Mock auth lockout state (5-attempt, 30-min cooldown) lives in `apps/storefront/src/lib/auth/mock.ts`. To reset mid-test without a server restart, clear the relevant `localStorage` key or hard-refresh; verify the actual mechanism in `mock.ts`.
- The forgot-password success copy must echo the submitted email verbatim AND must render identically for known and unknown emails. Any deviation is a P1 information-disclosure bug.
- The "Validating link…" indicator on `/reset-password` should not become the LCP element — relevant for TASK-007 (Performance), not gated here, but worth noting if observed.
- Real Medusa session creation, JWT cookies, brute-force backend lockout, and reset-token storage in Medusa are **out of scope** — those are covered by TASK-004 (Integration), TASK-009 (Security), and TASK-100 (QA).
- Brand theme tokens live in `apps/storefront/src/app/globals.css`. Mulish font is wired via `next/font`. `lh3.googleusercontent.com` is allow-listed in `next.config.ts` `remotePatterns`.
- Use the React DevTools "Components" tab to confirm which components are client (have hooks) vs server (no hooks). Page-level components in `app/(auth)/*/page.tsx` should all be server components.
