## Description
End-to-end test of the Treasure Trove storefront slide-in Cart Drawer against the mock cart layer. Covers the full cart-from-anywhere flow — opening the drawer from a PDP add-to-cart (or the global CartTrigger), the Framer Motion slide-in panel from the right, the dimmed backdrop, close-on-backdrop and close-on-ESC, the CartItem row (product image, title, variant, quantity stepper, remove icon, line total), the CartSummary (subtotal, shipping note, estimated total, "Proceed to Checkout" CTA), the EmptyCart state with "Continue Shopping" link back to `/products`, the per-row line-item Select checkbox and Select-All toggle (driving the checkout total + CTA enable/disable), the two required edge-case states inside CartItem — Out of Stock (badge + disabled quantity controls + warning row) and Price Changed Since Added (struck-through old price + new price + inline warning) — plus code-standard checks (no raw `<img>`, no real Medusa traffic from mock paths, `data-testid` coverage on every interactive element, drawer mounted via the root layout) and responsive layout across 360 / 768 / 1440 px viewports (full-height on mobile, 420 px wide on desktop). Cart open/close and all line-item operations are driven by the Zustand store at `@/stores/cart.ts`.

---

## Preconditions
- `apps/storefront` is running (`pnpm dev --filter=storefront` at `http://localhost:3000`)
- The Zustand cart store at `src/stores/cart.ts` is wired and exposes `addItemLocal`, `removeItem`, `updateQuantity`, `openCart`, `closeCart`, `toggleSelected`, `selectAll`, `deselectAll`
- Mock cart data at `src/lib/cart.mock.ts` (or the live mock seed used by `addItemLocal`) is available; the dataset can produce:
  - At least one in-stock CartLineItem (image, title, variant label, unit price, line total)
  - At least one item flagged as Out of Stock (`available: false` / `inventory: 0`) for the OOS row test
  - At least one item flagged with a price change since added (old price + new price for the Price Changed warning test)
- `CartDrawer` is mounted in the root layout (`apps/storefront/src/app/layout.tsx`) so it is available on every page
- A working PDP route (`/products/halden-brass-pendant-lamp`) is reachable to seed the cart via `addItemLocal`
- Browser in a fresh state — no cookies, no `localStorage` keys carrying cart state from prior runs
- Browser DevTools open with the Network tab visible (so you can confirm no real Medusa requests fire during mock-driven steps)
- Chrome DevTools device toolbar available for mobile / tablet / desktop viewports
- Browser does NOT have `prefers-reduced-motion: reduce` set system-wide (the slide-in transition is validated explicitly)

## Expected Result
The Cart Drawer behaves end-to-end against the Zustand-driven mock cart. Opening from `cart-trigger` (or a PDP add-to-cart) slides the panel in from the right with a smooth Framer Motion transition and a dimmed backdrop; clicking the backdrop, pressing ESC, or clicking the close X all close it. Each CartItem row renders image, title, variant, quantity stepper (decrement disabled at 1), unit price, line total, a per-row select checkbox (auto-checked on add), and a remove icon. The select-all toggle and individual checkboxes drive what gets summed in `cart-subtotal` / `cart-total`, and disable the `checkout-cta` when nothing is selected. "Proceed to Checkout" navigates to `/checkout`. The EmptyCart state renders inside the drawer when no items remain, with a "Continue Shopping" link back to `/products`. The Out-of-Stock row shows the OOS badge, disables quantity controls, and is excluded from totals; the Price-Changed row shows the old price struck through, the new price highlighted, and an inline warning. The drawer is full-height on mobile and 420 px wide on desktop; focus is trapped inside while open and returns to the trigger on close. All interactive elements expose `data-testid` attributes, every image renders via `next/image`, and the Network tab confirms no real Medusa traffic for mock-driven cart operations. `CartDrawer` is mounted once via the root layout and available on every route.

## Notes
- The current `useCartStore` at `src/stores/cart.ts` exposes BOTH a real Medusa-backed path (`addItem`, `removeItem`, `updateQuantity` → `*Action` server actions) AND a mock-safe `addItemLocal`. This TC is the **mock-safe** scope — `addItemLocal` and direct store mutations only. Any step that incidentally exercises the real `addItem` / `removeItem` / `updateQuantity` paths must NOT issue a `localhost:9000` request because mock mode should short-circuit before the action fires; if it does, file a P0 bug against the mock wiring (real-vs-mock cart sync is TASK-036 Integration scope).
- Newly-added items are **auto-selected** by design — `addItemLocal` pushes the new id onto `selectedIds` and the `syncSelection` helper preserves selection across reconciliations. Any test step where a fresh add appears unchecked is a P1 regression.
- Quantity stepper minimum is **1** — full removal is the trash icon's job. Any path that lets the stepper reach 0 (which would silently drop the line) is a P1 bug.
- `cart-select-all` may render an **indeterminate** state when some lines are selected and some are not — confirm via the checkbox's `indeterminate` DOM property, not just its visual rendering.
- The CartDrawer must be mounted **once** at the root layout level. If you find `data-testid="cart-drawer"` rendered more than once in the DOM (e.g. duplicated per route), it is a P1 bug — multiple stores will race for the same Zustand state and the close-on-backdrop logic will misbehave.
- The Price-Changed and Out-of-Stock edge cases are **driven by the line item's own data fields** (`available`, `inventory`, and the price-delta detection in `CartItem.tsx`) — there is no global "OOS mode" toggle. Seed each edge case at the line level, not at the store level.
- Real Medusa cart creation, server-side line-item sync, payment-region pricing, and the `/checkout` flow itself are **out of scope** — those are covered by TASK-035 (Backend), TASK-036 (Integration), and TASK-041 (Security).
- Use the React DevTools "Components" tab to confirm that `CartDrawer`, `CartItem`, `CartSummary`, and `EmptyCart` are all client components (they read the Zustand store), while the page-level layout that mounts `CartDrawer` may itself be a Server Component.
