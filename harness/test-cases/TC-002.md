## Description
End-to-end test of the Treasure Trove storefront homepage UI against mock data. Covers all six required sections in correct visual order — Hero → Marquee → Collections → Featured Products → Blog Preview → Brand Philosophy — plus the additional sections shipped in TASK-010 (HeroCarousel, TrustStrip, CategorySection, OfferCarousel, ShopByPrice, New Arrivals). Validates layout primitives (Navbar with full/island/hidden scroll modes, MobileHeader with cycling search placeholder, MobileSidebar drawer, BottomTabBar with cart badge, Footer with NewsletterForm), Framer Motion scroll-reveal animations, error fallbacks (products load error + CMS content unavailable), `next/image` usage, `data-testid` coverage, Server-Component-by-default rule, and mobile-first responsive layout across 360 / 768 / 1440 px viewports. No real Medusa or Payload calls — mock data layers (`src/lib/medusa.mock.ts`, `src/lib/payload.mock.ts`) are the active implementation.

---

## Preconditions
- `apps/storefront` is running (`pnpm dev --filter=storefront` at `http://localhost:3000`)
- Mock data layers are active — confirm `src/lib/medusa.mock.ts` and `src/lib/payload.mock.ts` are wired
- Mock data provides at least: hero content, ≥ 4 collections, ≥ 6 featured products, 3 blog post previews, brand philosophy content, marquee text array
- Browser in a fresh state — no session cookies, no `localStorage` keys from prior runs
- Browser DevTools open with the Network tab visible (so you can confirm no real backend traffic)
- Chrome DevTools device toolbar available to switch between mobile / tablet / desktop viewports
- Browser does NOT have `prefers-reduced-motion: reduce` set system-wide (one step below explicitly toggles it)

## Expected Result
The homepage renders all six required sections in the correct order plus the additional sections shipped in TASK-010, on top of mock data with zero real backend traffic. The Navbar honours full / island / hidden scroll modes; MobileHeader, MobileSidebar drawer, and BottomTabBar function correctly on mobile. Hero uses `next/image` with `priority` and `sizes`; the marquee scrolls and pauses on hover; the CollectionsGrid is responsive (1 / 2 / 4 columns); BlogPreview shows exactly 3 cards with the right metadata; BrandPhilosophy preserves aspect ratio. Framer Motion / CSS scroll-reveal animations trigger on viewport entry for CollectionsGrid, FeaturedProducts, and BlogPreview, and are correctly disabled when `prefers-reduced-motion: reduce` is set. The products-load error renders the graceful message, and the CMS-unavailable case shows a static fallback hero. The page itself is a Server Component; only animation/interactive child components carry `"use client"`. Every interactive element is keyboard-reachable, has a focus ring, and exposes a `data-testid` attribute. No raw `<img>` exists in the rendered DOM.

## Notes
- The page MUST stay a Server Component (`app/page.tsx`). Any `"use client"` at the page level is a P0 violation — only inner interactive components (carousels, NewsletterForm, SectionReveal wrappers) may be client.
- Mock data layers (`src/lib/medusa.mock.ts`, `src/lib/payload.mock.ts`) are the spec for TASK-010 — never modify them to make a test pass; if the implementation diverges, file a bug.
- Scroll-reveal animations use native CSS `animation-timeline: view()` with a `tt-fallback-fade` for older browsers. Verify in `apps/storefront/src/app/globals.css` (search for `tt-*` selectors).
- The hero's full-bleed image is the LCP candidate — it MUST have `priority` set on `next/image` to avoid lazy loading. Removing `priority` is a P1 perf regression.
- `data-testid` rule applies to every interactive element on the page, including carousel arrows and dots, "Add to Cart" placeholders, and the newsletter input/submit.
- Real Medusa product fetching, Payload content, and ISR webhook revalidation are **out of scope** for this TC — those are covered by TASK-012 (Integration). This TC strictly exercises the UI against mocks.
- If the Cart badge in BottomTabBar does not reflect the Zustand store, it's a cart-state bug — track separately, but flag from this TC.
- Brand tokens, font wiring (Mulish via `next/font`), and CloudFront domain allow-listing live in `globals.css`, `layout.tsx`, and `next.config.ts` respectively.
