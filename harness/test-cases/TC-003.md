## Description
End-to-end test of the Treasure Trove storefront Products Listing page against the mock Medusa layer. Covers the three listing routes (`/products`, `/collections`, `/collections/[handle]`), the responsive 4 / 2 / 1-column ProductGrid, the FilterSidebar (material checkboxes, collection checkboxes, min/max price range), the SortToolbar (Featured · Price asc · Price desc · Newest), Pagination + "Load More" controls, Breadcrumb, CategoryHero, SubcategoryChips, ProductCard rendering (image, title, INR price, collection badge, hover state), and the three required edge-case states — No Products Found, Products Load Error, and Price Range Error (min > max). Validates Zustand filter/sort store behaviour, URL `useSearchParams` sync via Suspense, ISR `revalidate = 60`, Server Component vs `"use client"` boundary, header offset, `data-testid` coverage, and `next/image` usage. No real Medusa traffic.

---

## Preconditions
- `apps/storefront` is running (`pnpm dev --filter=storefront` at `http://localhost:3000`)
- Mock Medusa layer is active — `src/lib/medusa.mock.ts` provides `listProducts`, `getProducts`, and `getCollectionProducts` with parameters for filters, sort, page, and pageSize
- Mock data provides at least: 20 products distributed across ≥ 3 materials and ≥ 3 collections; products span a price range wide enough to exercise the min/max slider
- Zustand filter store is wired (`apps/storefront/src/stores/filter.ts`)
- Browser in a fresh state — no cookies or `localStorage` carrying filter state from prior runs
- DevTools open with the Network tab visible — used to confirm no real Medusa requests fire
- Chrome DevTools device toolbar available for mobile / tablet / desktop viewports

## Expected Result
The Products Listing page renders correctly on all three routes (`/products`, `/collections`, `/collections/[handle]`) against mock data with zero real Medusa traffic. The grid is 4 / 2 / 1-column responsive; ProductCards show image, title, INR price, and collection badge; hover states work on desktop. The FilterSidebar applies material, collection, and price filters reactively through the Zustand store; the SortToolbar provides exactly four sort options that update the grid. URL `useSearchParams` is in sync with the store and hydrates on reload. Pagination and Load More advance correctly. All three required edge states render: No Products Found (empty), Products Load Error (with Retry CTA), and Price Range Error (min > max — invalid filter is not applied). The page itself is a Server Component with `revalidate = 60`; only interactive children carry `"use client"` and are wrapped in `<Suspense>` for `useSearchParams`. Every interactive element exposes a `data-testid` and is keyboard-reachable. No raw `<img>` tags exist in the rendered DOM.

## Notes
- `FilterSidebar`, `SortToolbar`, `SubcategoryChips`, and `Pagination` are `"use client"` components — they must be wrapped in `<Suspense>` on the page because they read `useSearchParams`. Removing the Suspense boundary will hard-fail the build in Next 15; if you see a Suspense-related runtime warning, file as a bug against this TC.
- The page itself (`app/products/page.tsx`) MUST stay a Server Component with `revalidate = 60`. Adding `"use client"` to the page is a P0 violation.
- Filter and sort state lives in `apps/storefront/src/stores/filter.ts` (Zustand). URL query params are the source of truth on reload; the store hydrates from `useSearchParams`. Any divergence between URL and store after reload is a bug.
- Min > max should be **caught client-side BEFORE invoking the mock** — the mock must NOT be called with an invalid filter. Verify by watching the Network tab (no fetch should fire for the invalid step).
- Header offset values are `pt-[104px]` mobile and `pt-[136px]` desktop. If a section visually clips behind the header, the offset is wrong.
- Real Medusa filter parameters, server-side filtering, and `getCollectionProducts` against a real API are **out of scope** here — covered by TASK-020 (Integration). This TC strictly exercises the UI against mocks.
- The `data-testid` rule extends to every checkbox, input, button, sort dropdown option, pagination control, and product card.
- If the No Products Found CTA does not clear the URL query params (only the store), the next reload will reapply the empty filter and the user sees the empty state again — flag as a bug.
